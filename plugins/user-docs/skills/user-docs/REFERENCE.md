# User Documentation Reference

Comprehensive checklists, standards, and best practices for end-user documentation.

## Universal Documentation Standards

### Plain Language Requirements

All user-facing documentation must meet these readability standards:

- **Reading Level:** 8th grade or below (Flesch-Kincaid Grade Level ≤ 8)
- **Reading Ease:** Flesch Reading Ease score ≥ 60
- **Average Sentence Length:** Under 20 words
- **Average Word Length:** Under 5 characters (where possible)
- **Passive Voice:** Less than 10% of sentences

**Tools to check:**
- Hemingway Editor (hemingwayapp.com)
- Grammarly
- Readable.com
- Microsoft Word readability statistics

### Writing Style Standards

**Voice & Tone:**
- Second person ("you") for instructions
- Active voice (minimum 90% of sentences)
- Present tense
- Friendly, encouraging, professional
- Gender-neutral language

**Sentence Structure:**
- One idea per sentence
- Maximum 20 words per sentence (ideal: 15-17)
- Short paragraphs (2-4 sentences)
- Concrete, specific language

**Formatting:**
- Headers every 2-3 paragraphs
- Bulleted or numbered lists for scanability
- Bold for UI elements ("Click **Save**")
- Code formatting for technical terms (`filename.txt`)
- Ample white space

## Complete Documentation Quality Checklist

### Content Quality

- [ ] **Clear Purpose**: Title immediately tells user what they'll learn
- [ ] **User-Focused**: Organized by task, not by feature
- [ ] **Plain Language**: 8th grade reading level or below
- [ ] **Active Voice**: Minimum 90% of sentences
- [ ] **Concise**: No unnecessary words or fluff
- [ ] **Accurate**: All information verified and current
- [ ] **Complete**: No missing steps or assumed knowledge
- [ ] **Tested**: Instructions followed exactly by tester

### Structure & Organization

- [ ] **Logical Flow**: Information in sensible order
- [ ] **Progressive Disclosure**: Simple concepts before complex
- [ ] **Scannable**: Headers, bullets, white space
- [ ] **Consistent Format**: Same structure for similar content
- [ ] **Clear Hierarchy**: H1 > H2 > H3 structure makes sense
- [ ] **Table of Contents**: For articles over 1000 words
- [ ] **Related Links**: Cross-links to relevant articles

### Visual Elements

- [ ] **Screenshots Included**: For every UI interaction
- [ ] **Screenshots Current**: Match latest product version
- [ ] **Screenshots Annotated**: Arrows, boxes, or numbers highlight key elements
- [ ] **Screenshots High-Quality**: Clear, properly sized, consistent style
- [ ] **Alt Text**: Descriptive alt text for all images
- [ ] **Diagrams/Videos**: For complex multi-step processes (when helpful)
- [ ] **Consistent Style**: Same annotation colors, fonts, window sizes

### User Experience

- [ ] **Time Estimate**: Provided for tutorials and guides
- [ ] **Prerequisites**: Listed clearly at the beginning
- [ ] **Success Criteria**: User knows when they've succeeded
- [ ] **Error Handling**: Common problems addressed
- [ ] **Next Steps**: Links to related content
- [ ] **Searchable**: Uses keywords users actually search for
- [ ] **Mobile-Friendly**: Readable on phones and tablets

### Technical Quality

- [ ] **Links Work**: All internal and external links tested
- [ ] **No Broken Images**: All images load correctly
- [ ] **Proper Formatting**: No formatting errors (extra spaces, wrong fonts)
- [ ] **Spell Check**: No typos or grammatical errors
- [ ] **Version Noted**: Documentation version/date indicated
- [ ] **Metadata**: Proper tags, categories, descriptions for SEO

## Documentation Type Guidelines

### Tutorial / How-To Article

**Purpose:** Teach user to complete specific task

**Length:** 500-1500 words, 5-15 steps

**Required Elements:**
- [ ] Task-based title: "How to..."
- [ ] Time estimate (e.g., "10 minutes")
- [ ] Prerequisites clearly listed
- [ ] What user will accomplish stated upfront
- [ ] Step-by-step numbered instructions
- [ ] One action per step
- [ ] Screenshot after each major step
- [ ] Visual confirmation of success
- [ ] "What you accomplished" summary
- [ ] Next steps / related tutorials

**Best Practices:**
- Start with the simplest path to success
- Include only essential steps (no tangents)
- Show expected results after each step
- Address one common error/issue
- Test by having someone follow exactly

---

### Knowledge Base Article

**Purpose:** Answer specific question or solve specific problem

**Length:** 300-800 words

**Required Elements:**
- [ ] Question-based title
- [ ] Quick answer in first paragraph
- [ ] Detailed explanation
- [ ] Step-by-step instructions (if applicable)
- [ ] Screenshots or examples
- [ ] "Related Articles" section
- [ ] Last updated date
- [ ] Helpfulness rating widget

**Best Practices:**
- Answer the question immediately
- Use the exact question users ask (check support tickets)
- Link to related articles
- Update regularly based on user feedback
- Track analytics (views, helpfulness rating)

---

### User Guide (Comprehensive)

**Purpose:** Complete reference for feature or product

**Length:** 20-100+ pages (if PDF) or multi-page site

**Required Elements:**
- [ ] Table of contents with page numbers/links
- [ ] Getting started section (first 10-15 pages)
- [ ] Overview/introduction to product
- [ ] Core concepts explained
- [ ] Feature-by-feature coverage
- [ ] Common tasks section
- [ ] Troubleshooting section
- [ ] Index (for PDF) or search (for web)
- [ ] Version number and date

**Structure:**
1. Introduction & Overview (5%)
2. Getting Started (15%)
3. Core Features (40%)
4. Common Tasks (20%)
5. Advanced Features (10%)
6. Troubleshooting (5%)
7. Reference (5%)

**Best Practices:**
- Design for both linear reading and reference
- Include lots of cross-references
- Update with each product release
- Offer both web and PDF versions
- Create separate beginner and advanced sections

---

### Quick Reference / Cheat Sheet

**Purpose:** Quick lookup for common actions

**Length:** 1-2 pages maximum

**Required Elements:**
- [ ] Most common actions only
- [ ] Table or grid format
- [ ] Keyboard shortcuts (if applicable)
- [ ] Brief descriptions (1 sentence max)
- [ ] Printable format (PDF)
- [ ] Visual grouping of related items

**Best Practices:**
- Include only the essentials (top 20%)
- Use table format for scannability
- Make it truly printable (fits on 1-2 pages)
- Use consistent formatting
- Update when workflows change

---

### FAQ

**Purpose:** Quick answers to common questions

**Length:** 50-200 words per answer, 10-50 total questions

**Required Elements:**
- [ ] Actual user questions (not assumed)
- [ ] Organized by category/topic
- [ ] Expandable/collapsible format (accordion)
- [ ] Search functionality
- [ ] Links to detailed articles
- [ ] "Still have questions?" CTA

**Best Practices:**
- Use real questions from support tickets
- Order by most frequently asked
- Keep answers brief (link to KB for details)
- Review and update quarterly
- Add new questions as they arise
- Remove outdated questions

---

### Troubleshooting Guide

**Purpose:** Help users diagnose and fix problems

**Length:** Variable, problem-dependent

**Required Elements:**
- [ ] Symptom-based organization
- [ ] "Quick fixes" section at top
- [ ] Common issues first (ordered by frequency)
- [ ] Clear problem descriptions
- [ ] Step-by-step solutions
- [ ] Multiple solutions when applicable
- [ ] Error message explanations
- [ ] Escalation path (when to contact support)

**Structure:**
1. Quick Fixes (try these first)
2. Common Issues (by symptom or error)
3. Error Messages Explained
4. Browser/Platform-Specific Issues
5. Still Having Trouble? (contact info)

**Best Practices:**
- Use actual error messages as headers
- Provide multiple solution methods
- Include prevention tips
- Link to related KB articles
- Update based on support trends

---

### Onboarding / Getting Started

**Purpose:** Help new users reach first success quickly

**Length:** Short (500-1000 words, 5-10 minutes)

**Required Elements:**
- [ ] Welcome message
- [ ] Product overview (2-3 sentences)
- [ ] Time estimate
- [ ] Goal clearly stated
- [ ] Minimum steps to success
- [ ] Encouraging tone throughout
- [ ] Visual confirmation of success
- [ ] Next steps clearly stated

**Best Practices:**
- Minimize setup steps
- Get to value as fast as possible
- Show tangible result
- Use encouraging language
- Don't overwhelm with options
- Include completion checklist
- Celebrate user's success

---

### Release Notes (User-Facing)

**Purpose:** Inform users of new features and improvements

**Length:** 200-500 words

**Required Elements:**
- [ ] Date of release
- [ ] New features section (with benefits)
- [ ] Improvements section
- [ ] Bug fixes section (user-facing only)
- [ ] Screenshots or GIFs of new features
- [ ] Links to detailed documentation
- [ ] What's coming next (optional)

**Best Practices:**
- Focus on user benefits, not technical details
- Show, don't just tell (screenshots/GIFs)
- Use friendly, excited tone
- Avoid technical jargon
- Link to tutorials for new features
- Keep it scannable
- Send via email and publish to blog/changelog

## Screenshot Standards

### Technical Requirements

**Image Format:**
- PNG for UI screenshots (supports transparency)
- JPG for photos (smaller file size)
- GIF or MP4 for animations

**Resolution:**
- Minimum 1280px width for desktop screenshots
- 2x resolution for Retina displays (recommended)
- Mobile screenshots: actual device resolution

**File Size:**
- Optimize for web (under 500KB per image)
- Use compression tools (TinyPNG, ImageOptim)

**Naming Convention:**
```
[feature]-[action]-[step-number].png
Examples:
- dashboard-create-project-1.png
- settings-add-user-2.png
- login-reset-password.png
```

### Content Standards

**What to Include:**
- Only relevant UI elements (crop tightly)
- Default/clean state when possible
- Realistic but non-sensitive data
- Current product version only

**What to Exclude:**
- Personal information (names, emails, addresses)
- Browser bookmarks and tabs
- Desktop clutter
- Operating system UI (taskbar, dock)
- Outdated UI elements

### Annotation Standards

**Colors:**
- Red for primary highlights (#FF0000 or #E74C3C)
- Blue for secondary information (#3498DB)
- Yellow for warnings (#F39C12)
- Consistent throughout all documentation

**Annotation Types:**
- **Red box/rectangle**: Highlight clickable button or UI element
- **Red arrow**: Point to specific location or element
- **Numbered circles**: Show sequence of steps (1, 2, 3)
- **Text callouts**: Add brief explanatory text
- **Blur/pixelate**: Hide sensitive information

**Best Practices:**
- Use annotations sparingly (don't clutter)
- Be consistent with style (same colors, line widths)
- Make annotations obvious but not overwhelming
- Use numbers for multi-step processes
- Add arrow pointing to what user should click

### Screenshot Checklist

Before including a screenshot:

- [ ] **Current**: Shows latest product version
- [ ] **Cropped**: Only relevant UI visible
- [ ] **Annotated**: Key elements highlighted
- [ ] **High Quality**: Clear, not pixelated
- [ ] **Optimized**: File size under 500KB
- [ ] **Alt Text**: Descriptive text for screen readers
- [ ] **Privacy**: No personal/sensitive information
- [ ] **Consistent**: Same style as other screenshots
- [ ] **Named**: Descriptive filename

## Accessibility Standards

### Images & Media

- [ ] **Alt Text**: Every image has descriptive alt text
- [ ] **Transcripts**: Videos include text transcripts
- [ ] **Captions**: Videos have closed captions
- [ ] **Audio Descriptions**: Complex visuals described in text
- [ ] **Color**: Don't rely on color alone to convey information

### Text & Structure

- [ ] **Heading Hierarchy**: Proper H1 > H2 > H3 structure
- [ ] **Link Text**: Descriptive ("View tutorial" not "click here")
- [ ] **List Formatting**: Use proper list markup (ul, ol)
- [ ] **Tables**: Include headers and captions
- [ ] **Contrast**: Text has sufficient contrast with background

- [ ] **Keyboard Navigation**: All interactions work with keyboard only
- [ ] **Focus Indicators**: Clear visual focus on interactive elements
- [ ] **Skip Links**: "Skip to content" link provided
- [ ] **Breadcrumbs**: Show user's location in site hierarchy

## SEO & Findability

### Title Optimization

**Format:**
```
[Task/Question] | [Product Name]
```

**Examples:**
- How to Reset Your Password | Example App
- What is Two-Factor Authentication? | Example App
- Troubleshooting Login Issues | Example App Help

**Best Practices:**
- Use exact phrases users search for
- Include primary keyword
- Keep under 60 characters
- Make it descriptive and specific

### Meta Description

**Length:** 120-160 characters

**Format:**
```
Learn how to [task] in [product]. [Additional context]. Step-by-step guide with screenshots.
```

**Example:**
> Learn how to create your first project in Example App. This quick guide takes 5 minutes and includes screenshots for each step.

### Keywords

**Primary Keyword:**
- Main topic/task of the article
- Use in title, H1, first paragraph, and conclusion
- Use naturally, don't force it

**Secondary Keywords:**
- Related terms and variations
- Use in H2 headings and throughout body
- Include common misspellings and synonyms

### Internal Linking

- Link to related articles (3-5 per article)
- Use descriptive anchor text
- Link to higher-level overview pages
- Build topic clusters (hub and spoke model)

### URL Structure

**Good URLs:**
```
/help/getting-started/create-account
/help/projects/how-to-create-project
/help/troubleshooting/login-issues
```

**Bad URLs:**
```
/help/article/12345
/doc?id=abc123
/help/page.html
```

**Best Practices:**
- Use descriptive, keyword-rich URLs
- Use hyphens, not underscores
- Keep URLs under 75 characters
- Use lowercase only
- Match page title/heading

## Maintenance & Updates

### Update Frequency

**Critical (Update Immediately):**
- UI changes that make screenshots outdated
- Feature deprecations or removals
- Broken links or incorrect information
- Security-related content

**High Priority (Update Within 1 Week):**
- New feature releases
- Workflow changes
- Error message changes
- Common user questions not covered

**Regular (Update Monthly/Quarterly):**
- Minor improvements
- Additional examples
- Expanded explanations
- SEO optimization

### Version Control

**Track These Changes:**
- Last updated date (always visible)
- Version number (for major updates)
- Change log (what was updated)
- Author/reviewer information

**Best Practices:**
- Review all docs quarterly
- Archive outdated content (don't delete)
- Use version control (Git)
- Track page analytics to prioritize updates
- Set up alerts for UI changes

### Deprecation Process

When removing or replacing documentation:

1. **Mark as Deprecated**: Add notice at top of article
2. **Link to Replacement**: Direct users to new/updated content
3. **Keep Live for 6 Months**: Give users time to transition
4. **Redirect to New Content**: Set up permanent redirect (301)
5. **Archive (Don't Delete)**: Keep for reference

**Deprecation Notice Example:**
```markdown
> ⚠️ **This article is outdated.** We've updated this feature. See the [new guide](link) for current instructions.
```

## Analytics & Metrics

### Track These Metrics

**Usage Metrics:**
- Page views (total and unique)
- Time on page (average)
- Bounce rate
- Search queries leading to page
- Internal search queries (what users search for)

**Engagement Metrics:**
- Helpfulness ratings (thumbs up/down)
- Comments or feedback
- Links clicked (which next steps users choose)
- Video play rate (for embedded videos)

**Quality Indicators:**
- Support ticket volume (does good docs reduce tickets?)
- Search-to-page ratio (how many searches to find page?)
- Exit rate (do users leave satisfied?)
- Scroll depth (do users read the whole article?)

### Use Analytics to Improve

**High Views + Low Helpfulness = Needs Improvement**
- Content exists but isn't solving the problem
- Rewrite with focus on user's actual question
- Add more screenshots or examples

**High Views + High Bounce = Wrong Content**
- Title is misleading (doesn't match content)
- Users expected something different
- Revise title to be more accurate

**Low Views + High Helpfulness = Findability Problem**
- Content is good but hidden
- Improve SEO (title, keywords, links)
- Add to navigation
- Link from related articles

**High Support Tickets = Missing Docs**
- Users can't find answer in docs
- Create new article addressing the question
- Improve existing article

## Quality Rubric

### Excellent (5/5)

- Reading level: Grade 6-8
- All steps tested and verified
- Screenshots for every UI interaction
- Clear, specific, actionable language
- Perfect grammar and spelling
- Engaging, encouraging tone
- Comprehensive troubleshooting
- Updated within last month
- High user ratings (90%+ helpful)

### Good (4/5)

- Reading level: Grade 8-10
- Steps verified
- Screenshots for main steps
- Clear language, minor jargon
- Few grammar/spelling errors
- Professional tone
- Basic troubleshooting
- Updated within last 3 months
- Good user ratings (75-89% helpful)

### Adequate (3/5)

- Reading level: Grade 10-12
- Steps not fully tested
- Some screenshots
- Some unclear language
- Several grammar/spelling errors
- Neutral tone
- Minimal troubleshooting
- Updated within last 6 months
- Average ratings (60-74% helpful)

### Needs Improvement (2/5)

- Reading level: Above grade 12
- Steps not tested
- Few screenshots
- Confusing or vague language
- Many errors
- Overly technical tone
- No troubleshooting
- Outdated (6-12 months)
- Low ratings (40-59% helpful)

### Poor (1/5)

- Very high reading level
- Steps incomplete or wrong
- No screenshots
- Unclear or incorrect
- Numerous errors
- Inappropriate tone
- No support resources
- Very outdated (1+ years)
- Very low ratings (<40% helpful)

## Style Guide Quick Reference

### Capitalization

- **Sentence case for headings**: "How to create a project" not "How to Create a Project"
- **UI elements**: Match exact capitalization ("Save" if button says Save)
- **Product names**: Follow brand guidelines (e.g., "iPhone" not "Iphone")

### Punctuation

- **Serial comma**: Use it (red, white, and blue)
- **Periods in lists**: If list items are complete sentences
- **No periods in UI labels**: "Save" not "Save."

### Numbers

- **Spell out one through nine**: "five users" not "5 users"
- **Use numerals for 10+**: "12 projects" not "twelve projects"
- **Always numerals in technical contexts**: "3 GB" not "three GB"

### Abbreviations

- **Spell out first use**: "Application Programming Interface (API)"
- **Common abbreviations OK**: FAQ, URL, PDF (don't spell out)
- **Avoid Latin abbreviations**: Use "for example" not "e.g."

### UI Elements

- **Bold for clickable items**: "Click **Save**"
- **Code formatting for technical**: Type `filename.txt`
- **Angle brackets for nested menus**: Go to **Settings > Preferences > General**

## Glossary of Terms

### Documentation Types

- **Tutorial**: Step-by-step guide to complete specific task
- **How-To**: Same as tutorial
- **Guide**: Comprehensive reference for feature or topic
- **Knowledge Base (KB) Article**: Answer to specific question
- **FAQ**: Frequently Asked Questions
- **Quick Reference**: Brief lookup sheet (1-2 pages)
- **Cheat Sheet**: Same as quick reference
- **Troubleshooting Guide**: Help for diagnosing and fixing problems
- **Release Notes**: What's new in latest release

### Writing Concepts

- **Active Voice**: Subject performs action ("Click Save" not "Save should be clicked")
- **Passive Voice**: Action performed on subject (avoid in instructions)
- **Plain Language**: Simple, clear, everyday words
- **Jargon**: Technical or specialized terms (avoid or explain)
- **Flesch Reading Ease**: Readability score (60+ is good for user docs)
- **Grade Level**: Reading comprehension level (aim for grade 8 or below)
- **Second Person**: Addressing reader as "you"

### User Experience

- **Progressive Disclosure**: Present simple first, complex later
- **Scannable**: Easy to skim (use headers, bullets, white space)
- **Information Architecture**: How content is organized
- **User Journey**: Path user takes through documentation
- **Findability**: How easily users can find what they need

---

**Remember:** Great user documentation is:
1. **Clear** - Uses plain language anyone can understand
2. **Visual** - Shows with screenshots, not just text
3. **Tested** - Verified by following instructions exactly
4. **Current** - Always up-to-date with latest product version
5. **Empathetic** - Written from user's perspective, solves their problems
